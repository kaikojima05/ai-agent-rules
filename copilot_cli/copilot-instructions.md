# 規約
## 実装の流れ
1. テストファイルにテストシナリオだけを書く（日本語）
2. 一度手を止めて、ユーザーにテストシナリオを確認・修正してもらう
3. テストシナリオをもとに、TDDのサイクルを回す
4. 1~3を繰り返す


## TDD必須
実装コードの新規作成・編集時は以下（TDD）を厳守すること。

1. 編集対象のファイルと同階層に .test ファイルがなければ作成する（以後、作成したファイルを使用する）
2. 失敗するテストを書き、失敗を確認する（Red）
3. テストを通す最小限の実装を書く（Green）
4. テストを維持しながらリファクタリングする（Refactor）

テストなしの実装、順序の省略は禁止。1テスト1アサーションを原則とする。

## 命名規則
1. 変数名、関数名、クラス名は既存の実装パターンに合わせる

## 関数、変数、クラスの設計
1. 単一責任原則とする
2. if や for が2層以上ネストする場合は、ローカルにヘルパー関数として処理を切り出すこと
3. 条件式内に三つ以上の条件がある場合は、ローカルにヘルパー関数として処理を切り出すこと
  a. ヘルパー関数には、先頭に必ず関数の目的をコメントすること
```typescript
// ダメな例
// props の値が条件を満たすかどうかを判定する関数
// details 中から必要なデータを filter() して sort() する関数

// 良い例
// 顧客データが有効かどうかを判定する関数
// 明細データの中から割引データを除き、金額の高い順に並び替える関数
```
4. 複数ファイルに跨って同一の処理がある場合は、編集対象のファイルと同階層に utils.ts を作成し、ヘルパー関数を追加すること
5. 戻り値の型は極力推論させること
  a. ただし、戻り値の型が分岐する場合は、明示的に戻り値の型を定義すること
6. 関数を追加・編集（rename）する際は、命名案をユーザーに複数個提案し、ユーザーが許可した名前で実装すること
7. マジックナンバーの使用は禁止
8. if や for が2層以上ネストする場合は、ローカルにヘルパー関数として処理を切り出すこと
  a. ただし、複数ファイルにまたがって同様の処理がある場合は、編集対象の schema.ts と同階層に utils.ts を作成し、ヘルパー関数自体を切り出すこと
9. dayjs を使用するときは @front/modules/dayjs から呼び出すこと
  a. ただし、@front/modules/dayjs に存在する dayjs の使用は非推奨としているので、ファイル内に用意された適切な関数を使用する
10. 新しく定義する関数やコンポーネントはファイルの最終行に追加すること

## 出力スタイル
1. 一度に10行以上のコードを追加・変更を行うことは禁止※テストコードは除く
2. コードの追加・変更時に必ず理由をユーザーに説明すること※テストコードは除く
3. ユーザーに説明を行う際に、変数名や関数名で意図を簡略化して伝えることは禁止、必ず本質的な説明を行うこと
