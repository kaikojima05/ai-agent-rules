# 規約
## 実装フロー
1. テストファイルにテストシナリオだけを書く（日本語）
2. 一度手を止めて、ユーザーにテストシナリオを確認・修正してもらう
3. テストシナリオをもとに、TDDのサイクルを回す
4. 1~3を繰り返す

## 実装規約
1. 単一責任原則
2. 関心の分離
3. インターフェースに対してプログラミングする
4. TDD 必須
  - 編集対象と同階層に `.test` ファイルがなければ先に作成する
  - Red → Green → Refactor の順序を厳守する
  - テストなしの実装、順序の省略は禁止
  - 1テスト1アサーションを原則とする
  - テストランナーは Vitest を使用する（Jest は使わない）
5. 可読性を優先し、宣言的な書き方を選ぶこと
  - 例: reduce() ではなく filter().map() を使う
  - Why: reduce() はループ中のアキュムレータの状態把握が難しく、レビュー負荷が高い
  - ただし、パフォーマンスが計測上ボトルネックになっている箇所には適用しない

## 命名規則
1. 変数名・関数名・クラス名は既存の実装パターンに合わせる
2. 関数の追加・rename 時は命名案を複数提案し、ユーザーの許可を得る

## 関数・変数・クラスの設計
### 遵守事項
1. if や for が2層以上ネストする場合は、ローカルに関数として処理を切り出す
2. 条件式内に三つ以上の条件がある場合は、条件式自体を変数として切り出すこと
3. 複数ファイルに跨って同一の処理がある場合は、編集対象のファイルと同階層に `utils.ts` を作成し、ヘルパー関数を追加すること
4. ヘルパー関数の作成時は、関数の先頭に処理内容をコメントで記述する
```typescript
// ダメな例
// props の値が条件を満たすかどうかを判定する関数
// details 中から必要なデータを filter() して sort() する関数

// 良い例
// 顧客データが有効かどうかを判定する関数
// 明細データの中から割引データを除き、金額の高い順に並び替える関数
```
5. 戻り値の型は極力推論させる
  - ただし、戻り値の型が分岐する場合は、明示的に戻り値の型を定義する
6. 新しく定義する関数やコンポーネントはファイルの最終行に追加する

### 禁止事項
1. マジックナンバーの使用
2. dayjs を `"dayjs"` から呼び出すこと
  - 必ず `"@/modules/dayjs"` に用意された関数を呼び出すこと
  - ただし、`"@/modules/dayjs"` の dayjs の使用は非推奨としているので、その他の適切な関数を使用する

## 出力制限
1. 一度に10行以上のコード追加・変更は禁止（テストコードは除く）
2. コードの追加・変更時は必ず理由をユーザーに説明する（テストコードは除く）
3. ユーザーに説明を行う際に、変数名や関数名で意図を簡略化して伝えることは禁止、必ず本質的な説明を行うこと
