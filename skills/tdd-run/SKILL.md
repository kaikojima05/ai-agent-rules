## 目的

`scenario` → `red` → `green` → `refactor-green` の 4 フェーズを **1 スキルに統合**し、人間の承認を 3 ゲートに絞って連続実行する。
「シナリオだけ書く」「実装だけ回す」といった部分利用も、別スキルを立てず引数で吸収する（下記「引数」）。

設計思想: **「テストシナリオさえ人間が確認・調整していれば、中のテストコードは読まなくてよい」** を前提に、
シナリオ確定後の実装ループ（テストを書く → 落ちる → 通す）は承認なしで走らせる。

## 承認ゲートは 3 つだけ

| ゲート | タイミング | 手段 |
|---|---|---|
| **GATE 1: シナリオ提案** | ループ開始前に 1 回 | `AskUserQuestion`（承認 / 調整） |
| **GATE 2: リファクタ提案** | 各サイクルの Green 後 | `AskUserQuestion`（適用するものを選択） |
| **GATE 3: 次テスト提案** | 各サイクルの末尾 | `AskUserQuestion`（次の 1 件を選択 / 終了） |

この 3 つ**以外**では `AskUserQuestion` を呼ばない。テスト本体実装・Red 確認・最小実装・Green 確認・リファクタ適用は
**報告（log）だけして自動で次に進む**。人間を待たせない。

## 引数

対象ファイルとは別に、フェーズ制御のキーワードを渡せる。

| 引数 | 挙動 |
|---|---|
| なし | フルフロー（GATE1 シナリオ提案 → 自動ループ） |
| `skip-implement` | シナリオだけ書いて GATE1 で停止。実装ループに入らない（旧 scenario の独立利用） |
| `skip-scenario` | GATE1 を飛ばし、既存の `it.todo` から自動ループを開始。`it.todo` が 1 件も無ければ停止 |

例: `/tdd-run src/services/billing/calculator.ts skip-implement`

## 実行フロー

### Step 0: 対象とテストランナーの確定

1. 引数でテスト対象ファイルを受け取る（無ければ一言だけ尋ねる）
2. `package.json` の `devDependencies` から Vitest / Jest を判定（`tdd-pattern.md` 準拠）
3. 同階層に `[ファイル名].test.ts(x)` が無ければ骨組みを Write する

### GATE 1: シナリオ提案（停止）

公開関数 / コンポーネントごとに正常系・境界値・異常系・副作用のシナリオ候補を
`it.todo`（日本語タイトル）で列挙し、`AskUserQuestion` で承認・調整を求める。

- describe のネストは 2 階層まで
- ここで人間が確定したシナリオが、以降の自動ループの**唯一の契約**になる
- 承認が出るまでループに入らない
- `skip-scenario` 指定時はこのゲートを飛ばし、既存の `it.todo` を契約として自動ループへ直行（`it.todo` が 1 件も無ければ停止）
- `skip-implement` 指定時はシナリオ確定後にここで停止し、自動ループへ進まない

### 自動ループ（GATE 2 / 3 以外は無停止）

確定した `it.todo` を 1 件ずつ処理する。**1 サイクル = 1 テスト**の粒度は崩さない。

#### a. テスト本体を実装（自動）

`it.todo('...')` を `it('...', ...)` にし、`Arrange / Act / Assert` で本体を書く。
1 テスト 1 アサーション。API テストはテスト用 DB を使う（Prisma モック禁止）。

#### b. テスト実行で Red 確認（自動・例外時のみ停止）

対象ファイルだけ実行し、**意図した失敗**であることを検証する。

- 「実装が無い / 期待値と異なる」での失敗 → 正常。**Red のエラー要約を log に出して**先へ進む
- syntax error / import error での失敗 → **テスト自身の構造バグ**。直前に自分で書いたテストの import 漏れ・タイポ・構文ミスなので、**構造だけ自己修正**して Red を取り直す（アサーションのロジックは変えない＝承認シナリオを維持）
  - 数回直しても解消しない場合のみ**例外停止**して人間に出す（無限ループの事故防止）
  - Why: テストを読まない運用でも構造エラーはランナー出力で箇所が分かり自己修正できる。人間を呼ぶのは直せない時だけでよい。ただしアサーションを書き換えると承認シナリオから逸脱するので、修正は構造に限定する

#### c. 最小実装（自動）

Red を通す最小限のコードだけ書く。ベタ書きで通るならベタ書きでよい（汎用化は GATE 2 の領分）。

ここでは行数や品質を問わない。コード量・構造の判断は GATE 2 の承認に委ねる（人間が提案を承認した時点で変更は理解済みであり、行数制限という代理指標を二重に課す必要がないため）。

#### d. テスト実行で Green 確認（自動）

対象テストが通り、かつ**他テストを壊していない**ことを確認。壊した場合は revert を提案して**例外停止**。

#### GATE 2: リファクタ提案（停止）

`function-pattern.md` / `ui-pattern.md` を毎回 Glob で拾い直して再読し、違反箇所だけを案として `AskUserQuestion` で提示する。

- 選ばれたものだけ Edit する。1 Edit = 1 観点
- **適用のたびにテスト再実行**し Green 維持を確認（これは自動）。崩したら即停止・revert 提案
- 「不要（リファクタなし）」も選択肢に必ず入れる
- `function-pattern.md` に反するリファクタは提案しない

#### GATE 3: 次テスト提案（停止）

残りの `it.todo`、および実装中に気づいた**新規シナリオ候補**を複数提示し、`AskUserQuestion` で次の 1 件を選ばせる。

- 「終了」を必ず選択肢に入れる。選ばれたらサイクルを抜けて完了報告
- 1 件選ばれたら、その 1 件で **a に戻る**
- ここで新ケースを足せるので、GATE1 で全部出し切れていなくてよい（テスト数を事前に決めなくてよい）

## 完了報告

GATE 3 で「終了」が選ばれたら、回したサイクル一覧（it・変更ファイル・適用したリファクタ）をまとめて報告し、
commit 準備を促す。

## 注意事項（「自動で走る」の境界線）

- **3 ゲート以外で `AskUserQuestion` を呼ばない。** 自動区間で確認を挟むと統合した意味が消える
- ただし以下の **事故は GATE と無関係に例外停止**する。3 ゲート哲学は「正常系では 3 つだけ」の意味であって「何があっても止まるな」ではない
  - Red の syntax / import error が自己修正でも解消しない（テストの構造バグ）
  - 他テストの破壊
  - `require-test.sh` 等の hook による deny
- **Red のエラー要約と Green の結果は、停止しなくても必ず log に残す。** 「読まなくていい」と「追えなくする」は違う。後から遡れるようにする
- テスト強制と自動化はどちらも本スキルの frontmatter hooks が担う。`require-test.sh` は**本スキル稼働中のみ**走り、対応テストの無いコード本体を deny する（テストファイルと対応テストありコードは通す）。その上で `allow-coding.sh` が対応テストありコードに allow を返して自動化する。**平時（ノースキル）は Edit/Write に hook が無く、通常の承認フローに委ねる**（テスト存在を門前払いの条件にしない）
- テスト実行コマンドは settings の allow が前提（未設定だと自動区間でも permission prompt が出て止まる）
- 本スキルは `tdd-pattern.md` の従属物。規約が変わったら本スキルより規約が優先
