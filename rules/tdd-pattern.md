# テストの編集ルール

## 実装フロー

1. テストファイルにテストシナリオだけを書く（日本語）
2. 一度手を止めて、ユーザーにテストシナリオを確認・修正してもらう
3. テストシナリオをもとに、TDD のサイクルを回す
4. 1〜3 を繰り返す

## テストランナー

1. プロジェクトに合わせて **Vitest** もしくは **Jest** を使用すること

## ファイル配置

1. 編集対象のファイルと同階層に `[ファイル名].test.ts(x)` を配置する
2. 編集対象と同階層に `.test` ファイルがなければ先に作成する

## 実装方針

1. TDD 必須
  - Red → Green → Refactor の順序を厳守する
  - テストなしの実装、順序の省略は禁止
2. 1 テスト 1 アサーションを原則とする
3. describe のネストは 2 階層までとする
4. API 関数のテストでは Prisma のモックではなくテスト用 DB を使用する
  - Why: 本物の DB 挙動と乖離したモックでは回帰を捕まえられないため

## テストの粒度

1. 結合テストを優先して書く
  - 外部依存をモックせず、テスト用 DB を通して振る舞い単位で検証する
  - Why: 実装の内部構造に縛られず、リファクタに強い回帰テストになるため
2. 次の関数は単体テストを書く
  - 外部 API を叩く関数 — モックして呼び出し条件・異常系を個別に検証する
  - 複雑な分岐ロジックを持つ関数 — 境界値・全分岐を結合テストで網羅するのは非効率なため
3. その他の関数は結合テストでカバーし、単体テストは後追いで足す
  - 結合テストで再現したバグは、回帰防止に単体テストを残す
  - 結合テストの粒度でデバッグに難航したロジックは、原因を単体で切り出す
  - 最初から網羅的な単体テストは書かない

## テスト用の export 禁止

1. テストしたいという理由だけで関数・変数を `export` してはならない
  - Why: 公開シンボルが増えると、読み手はそのコードを読むたびに「他から使われている可能性」を考慮させられ、認知負荷が上がるため
2. 非公開関数（複雑な分岐ロジックを持つヘルパー等）を直接テストしたくなっても、`export` せず
   公開 API 経由の結合テストでカバーする
  - Why: 「テストの粒度」の結合テスト優先と一致する。export を増やさずに振る舞いを検証できる

## 出力制限の例外

1. AGENTS.md の出力制限（30 行以上のコード追加・変更禁止、変更理由の説明義務）はテストコードには適用しない
  - Why: TDD のサイクルを止めないため
