## ガイドライン

### 開発手法

- **テスト駆動開発 (TDD)**: 原則としてTDDを実践すること。
  - テストファイルは、対象ファイルと同一ディレクトリ内に `[filename].test.ts(x)` の形式で作成すること。

### 設計原則

- **クリーンコードの遵守**: 以下の原則を徹底し、保守性の高いコードを記述すること。
  - **DRY** (Don't Repeat Yourself): 重複の排除。
  - **単一責任原則** (Single Responsibility Principle): 1つのモジュールは1つの役割に集中させる。
  - **再利用性** (Reusability): コンポーネントやロジックの再利用性を確保する。
  - **高凝集・疎結合** (High Cohesion & Loose Coupling): 関連する機能はまとめ、依存関係は最小限にする。
- **可読性の優先**: 複雑でトリッキーな実装よりも、意図が明確に伝わる可読性を最優先すること。

### コメント規約

- **「Why」を記述する**: コードそのものの説明（What）ではなく、設計意図や背景（Why）を記述すること。
- **日本語の使用**: コメントは必ず日本語で記述すること。

### 命名規則

#### ファイル・ディレクトリ

- **コンポーネント**: PascalCase (例: `UserProfile.tsx`)
- **Hooks**: `use` プレフィックスを用いた camelCase (例: `useSession.ts`)
- **ユーティリティ**: camelCase (例: `utils.ts`)
- **定数ファイル**: UPPER_SNAKE_CASE (例: `DATE_FORMAT.ts`)

#### 変数・関数・型

- **変数・関数・メソッド**: camelCase
- **定数**: UPPER_SNAKE_CASE
- **Reactコンポーネント / 型・Interface**: PascalCase

### インポート規則

- **パスエイリアスの利用**: プロジェクトルートからのインポートにはエイリアス `@/` を使用すること。
- **相対パスの制限**: 相対パスの使用は、同一ディレクトリ内の近接ファイルのみに限定すること。
- **インポート順序の推奨**:
  1. React / Next.js 関連のコアライブラリ
  2. 外部ライブラリ (Third-party)
  3. 内部モジュール (`@/` エイリアス)
  4. 相対パスによるインポート
  5. スタイル関連 (CSS/SCSS)

### スタイリング

- **カスタムクラスの優先**: プロジェクト固有の拡張クラス定義がある場合は、それを優先して使用すること。
- **条件付きレンダリング**: クラス名の動的な結合には `classnames` (または `clsx`) パッケージを使用すること。

### パッケージ管理

- パッケージ操作には必ず `Yarn` を使用すること。

## 禁止事項

- **責務の混在**: 1つの関数やコンポーネントに複数の役割（Responsibility）を持たせること。
- **過度な抽象化**: 意図が不明瞭になるほど抽象化レイヤーを増やすこと。
- **Whatの記述**: コードをなぞるだけのコメント（何をしているか）を書くこと。
- **過剰な共通化 (Over-engineering)**: 特定の1ファイルでしか使用されないロジックを、根拠なく共通 `utils` へ切り出す行為。
- **パッケージマネージャーの混在**: `npm` や `pnpm` の使用は禁止し、必ず `Yarn` を使用すること。
- **型エラーの放置**: TypeScriptの型エラーを無視すること。
- **any型やunknown型の乱用**: 安易に `any` や `unknown` 型を使用し、型安全性を損なうこと。
- **型アサーションの多用**: 不要な型アサーション (`as` キーワード) を多用すること。
